## Bug 1 — `if` instead of `assert` (Silent Failure)

**Location:** `test_user_profile_management()` — Verify update section

**The code:**
```python
if name_value == "John Doe":
    print("✓ Profile updated successfully")
```

**What's wrong:**  
Think of a test like a smoke detector. If the smoke detector sees smoke but only prints "smoke detected" on a piece of paper that nobody reads — the house still burns down. That's what `if` does here. If the value is NOT "John Doe", the test simply skips the print statement and keeps running. It never raises an error, so pytest marks the test as **PASSED** even though the profile update actually failed. This is a silent failure — the most dangerous kind of bug in testing.

**The fix:**
```python
assert name_value == "John Doe", f"Expected 'John Doe' but got '{name_value}'"
```
`assert` raises an error immediately if the condition is false, and the test is correctly marked as FAILED.

## Bug 2 — `get_attribute("value")` instead of `input_value()`

**Location:** `test_user_profile_management()` — Verify update section

**The code:**
```python
name_value = page.locator("#name").get_attribute("value")
```

**What's wrong:**  
Imagine a form field in HTML. When the page first loads, the field might have a default value written directly in the HTML code — for example `value="John"`. `get_attribute("value")` reads that original HTML value, not what the user actually typed. So even if the update failed and the field now shows something completely different on screen, this line would still return the old default value from the HTML — giving a false positive result.

**The fix:**
```python
name_value = page.locator("#name").input_value()
```
`input_value()` reads what is **currently displayed** in the input field, which is what we actually want to verify.

## Bug 3 — No wait after clicking Login

**Location:** `test_user_profile_management()` — Login section

**The code:**
```python
page.click("#login")
page.goto("https://example.com/profile")
```

**What's wrong:**  
Clicking the login button starts a process — the browser sends credentials to the server, the server validates them, and then redirects to the dashboard. This takes time. If the test immediately navigates to the profile page before login is complete, the server sees an unauthenticated request and likely redirects back to the login page. The test then tries to interact with a profile page that never actually loaded.

**The fix:**
```python
page.click("#login")
page.wait_for_url("**/dashboard**")  # Wait until login completes
page.goto("https://example.com/profile")
```

## Bug 4 — No wait before clicking `.confirm`

**Location:** `test_user_profile_management()` — Delete account section

**The code:**
```python
page.click(".delete-account")
page.click(".confirm")
```

**What's wrong:**  
Clicking "delete account" typically opens a confirmation modal/popup. But that popup takes a moment to appear — it's rendered by JavaScript after the click. If the test immediately tries to click `.confirm`, the element doesn't exist yet and Playwright throws an error. Even if it doesn't crash, it might be clicking the wrong element.

**The fix:**
```python
page.click(".delete-account")
page.wait_for_selector(".confirm", state="visible")  # Wait for modal to appear
page.click(".confirm")
```

## Bug 5 — `browser.close()` without `try/finally` (Resource Leak)

**Location:** `test_user_profile_management()` — end of function

**The code:**
```python
browser = p.chromium.launch()
# ... many test steps ...
browser.close()
```

**What's wrong:**  
If any step between `browser = p.chromium.launch()` and `browser.close()` throws an exception, Python immediately stops executing that function. This means `browser.close()` is never reached. The browser process keeps running in the background, consuming memory and system resources. Run this test 10 times with failures and you'll have 10 zombie browser processes.

**The fix:**
```python
browser = p.chromium.launch()
try:
    # ... test steps ...
finally:
    browser.close()  # This ALWAYS runs, even if an exception occurred
```

## Bug 6 — No assertion on API GET response

**Location:** `test_profile_api()` — Get profile section

**The code:**
```python
get_response = requests.get(f"https://api.example.com/profile/{user_id}")
print(f"Profile data: {get_response.text}")
```

**What's wrong:**  
The test makes a GET request to retrieve the profile it just created, but then only prints the response — it never actually checks if the data is correct. The GET request could return a 404 (not found), a 500 (server error), or even return a completely different profile — and the test would still pass because nothing is being verified. Printing is for debugging. Testing requires asserting.

**The fix:**
```python
get_response = requests.get(f"https://api.example.com/profile/{user_id}")
assert get_response.status_code == 200, "GET profile failed"
assert get_response.json()["name"] == "Test", "Profile name does not match"
```

## Bug 7 — Hardcoded Credentials (Security Issue)

**Location:** `test_user_profile_management()` — Login section

**The code:**
```python
page.fill("#username", "admin")
page.fill("#password", "admin123")
```

**What's wrong:**  
Passwords written directly in code get saved to Git history and are visible to anyone with access to the repository — forever, even if you delete them later. This is a serious security vulnerability. It also means that if the password changes, you have to find and update every place it appears in the code. In a real project, credentials like this could expose a production system.

**The fix:**
```python
import os
page.fill("#username", os.environ["TEST_USERNAME"])
page.fill("#password", os.environ["TEST_PASSWORD"])
```
Credentials are stored in environment variables or a secrets manager — never in code.

## Bug 8 — No Architecture (Everything in One Function)

**Location:** Entire file

**The code:**  
Both test functions contain all the logic inline — navigation, filling forms, clicking buttons, making API calls, and assertions — all mixed together with no structure.

**What's wrong:**  
Imagine a recipe book where every recipe also includes instructions for building the kitchen, sharpening the knives, and growing the vegetables. That's what this code looks like. There are no Page Objects, no separation between "how to interact with the page" and "what the test is verifying". This causes several practical problems:
- If a button selector changes, you have to find and fix it in every single test
- Tests are impossible to read — you can't quickly understand what a test is actually verifying
- Code cannot be reused between tests
- The tests violate POM (Page Object Model) — a fundamental principle of professional test automation

**The fix:**  
Create dedicated Page Object classes:
```python
class LoginPage:
    def login(self, username, password): ...

class ProfilePage:
    def update_name(self, name): ...
    def get_name(self): ...
    def delete_account(self): ...
```
Tests then become clean and readable:
```python
def test_update_profile(login_page, profile_page):
    login_page.login(USERNAME, PASSWORD)
    profile_page.update_name("John Doe")
    assert profile_page.get_name() == "John Doe"
```
